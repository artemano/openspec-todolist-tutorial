OpenSpec: un TodoList paso a paso

Antes de la primera sesión, cada estudiante necesita tener instalado:

Herramienta Para qué Verificación
Bun (≥ 1.1) Ejecuta la app y trae SQLite integrado (bun:sqlite), sin instalar nada extra de base de datos. bun --version
Git Clonar el repo base y guardar el historial de cambios. git --version
Claude Code — CLI o app de escritorio, cualquiera de las dos Es quien lee las specs y escribe el código. Aquí se escriben los comandos /opsx:*. claude --version o abrir la app
VS Code (o cualquier editor) Solo para leer los documentos que genera OpenSpec. No se escribe código. —

Cómo instalar cada herramienta

1 · Bun — un solo comando según tu sistema:

# macOS / Linux
curl -fsSL https://bun.sh/install | bash

# Windows (PowerShell)
powershell -c "irm bun.sh/install.ps1 | iex"

Cierra y vuelve a abrir la terminal, y verifica con bun --version.

2 · Git:

# macOS (instala las herramientas de línea de comandos)
xcode-select --install

# Linux (Debian/Ubuntu)
sudo apt install git

En Windows, descarga Git for Windows desde git-scm.com e instala con las opciones por defecto (deja marcado "Add Git to PATH"). Además de Git, esto le da a Claude Code la terminal Git Bash que usa internamente en Windows.

3 · Claude Code — requiere una cuenta con plan de pago (Pro, Max, Team o Console; el plan gratuito de Claude no lo incluye). El instalador oficial no necesita nada más (ni Node.js) y se actualiza solo:

# macOS / Linux / WSL
curl -fsSL https://claude.ai/install.sh | bash

# Windows (PowerShell, sin permisos de administrador)
irm https://claude.ai/install.ps1 | iex

Abre una terminal nueva, verifica con claude --version y ejecuta claude una primera vez: se abrirá el navegador para iniciar sesión con tu cuenta (solo la primera vez). Si algo falla, claude doctor diagnostica los problemas más comunes.

¿Prefieres la app de escritorio en vez de la terminal? Descárgala desde la página oficial de Claude Code e inicia sesión con la misma cuenta. El tutorial funciona igual con ambas — en el paso "El repo base" se explica el único detalle a tener en cuenta.

4 · VS Code — descarga el instalador para tu sistema desde code.visualstudio.com e instala con las opciones por defecto.

Importante: el estudiante no necesita saber React, SQL ni TypeScript. Su trabajo es describir, leer y aprobar. La IA construye.
✓ Verificación final: las tres verificaciones de la tabla responden con un número de versión, y claude abre sesión sin pedir login de nuevo. Con eso estás listo para el repo base.

Todo parte de un repositorio ya preparado (openspec-todolist-base) que se entrega. La idea es que descargue primero el repo base de GitHub de aquí.

Para ello realize desde la terminal:

git clone https://github.com/artemano/openspec-todolist-base.git

La constitución del proyecto: openspec/project.md

Este archivo ya viene escrito en el repo base y es el más importante: OpenSpec lo inyecta en todo lo que la IA genera, para que nunca "olvide" el stack. Contiene reglas como:

  • Frontend: React + Vite, componentes simples, sin librerías de UI externas.
  • Backend: Bun con bun:sqlite. Un solo archivo de base de datos: server/db.sqlite.
  • API: endpoints REST en JSON bajo /api/*.
  • Todo texto visible al usuario, en español.
openspec-todolist-base>
├── app/                  # Frontend React (Vite). Arranca con una pantalla vacía "TodoList".
├── server/               # API en Bun + bun:sqlite. Un endpoint /health de ejemplo.
│   └── db.sqlite         # Base de datos local. Un archivo, cero configuración.
├── .claude/
│   ├── skills/           # Skills preinstaladas: convenciones de React, API y SQLite,
│   │                     # más los skills de OpenSpec. La IA los lee automáticamente.
│   └── commands/opsx/    # Los comandos /opsx:* ya generados y commiteados.
│                         # Por eso el estudiante NUNCA ejecuta "openspec init".
├── openspec/
│   ├── project.md        # La "constitución" del proyecto (ya escrita, ver abajo).
│   ├── config.yaml       # Configuración de OpenSpec (generada, no se toca).
│   ├── specs/            # El plano maestro. Empieza vacío; crece con cada módulo.
│   └── changes/          # Aquí aparecerán las propuestas del estudiante.
└── package.json

Puesta en marcha

git clone <url-del-repo> openspec-todolist
cd openspec-todolist
bun install
bun dev        # abre http://localhost:5173 → pantalla base "TodoList"

¿Usas la app de escritorio de Claude Code (no la terminal)?

Funciona exactamente igual. Los comandos /opsx:* no viven en la app: viven en el repo (carpetas .claude/commands/opsx/ y .claude/skills/), y ya vienen incluidos en el repo base. Solo hay una regla:

  1. Abre la app y crea una nueva sesión seleccionando la carpeta del repo clonado (esto es lo crítico: los comandos son por-proyecto).
  2. Escribe /opsx en el chat: el autocompletado muestra propose, apply, archive…
  3. Sigue el tutorial normalmente. Donde diga "abre Claude Code", es tu sesión en la app.
Si los comandos no aparecen: pídele a Claude en el chat que ejecute openspec update (regenera los archivos de comandos) y abre una sesión nueva — los comandos se cargan al inicio de cada sesión.
✓ Checkpoint 0: la app abre en el navegador y muestra la pantalla base. Si esto no funciona, no continúes: pide ayuda aquí.

Cada módulo de este tutorial es exactamente el mismo ciclo. Al terminar el módulo 3 lo tendrás automatizado en la cabeza:

1 · /opsx:proposeDescribes el cambio. La IA genera el plano.
→
2 · Leer y corregirRevisas los documentos. Ajustas lo que no cuadre.
→
3 · /opsx:applyLa IA construye siguiendo las tareas.
→
4 · ProbarUsas la app y verificas.
→
5 · /opsx:archiveEl cambio pasa al plano maestro.
Regla de contexto: entre módulo y módulo, limpia el contexto del asistente (en Claude Code: /clear). OpenSpec funciona mejor con la mente fresca: todo lo que necesita saber está en las specs, no en el chat.

Objetivo del módulo: poder crear, listar, renombrar y eliminar proyectos. Es el módulo más guiado; los siguientes van soltando la mano.

Abre Claude Code en la carpeta del repo (claude) y escribe:

Copia y pega en Claude Code
/opsx:propose crear-proyectos

Quiero gestionar proyectos en el TodoList:
- Un proyecto tiene: nombre (obligatorio) y fecha de creación.
- Puedo: crear, ver la lista, renombrar y eliminar un proyecto.
- Al eliminar, pedir confirmación.
- UI: barra lateral izquierda con la lista de proyectos y un botón "Nuevo proyecto".
- Guardar todo en la base SQLite del repo.

La IA creará la carpeta openspec/changes/crear-proyectos/ con 4 documentos: proposal.md, specs/, design.md y tasks.md.

Este es el paso que separa esto del "vibe coding". Abre los documentos en VS Code y hazte estas preguntas:

Documento Pregunta
proposal.md ¿Entendió que quiero gestionar proyectos, no tareas todavía?
specs/ ¿Aparecen los 4 casos: crear, listar, renombrar, eliminar con confirmación?
design.md ¿Menciona una tabla projects en SQLite y endpoints /api/projects? (No hace falta entenderlo todo, solo que suene coherente con el pedido.)
tasks.md ¿Son pasos pequeños y concretos? ("Crear tabla", "Endpoint de listado", "Componente de barra lateral"…)

Ejercicio obligatorio: introduce un cambio a propósito. Dile a la IA en el chat:

Ejercicio: corregir el plano antes de construir
Actualiza la propuesta: los proyectos deben mostrarse ordenados del más reciente al más antiguo.

Verifica que la IA modificó los documentos (no el código, que aún no existe). Esa es la gracia: corregir aquí es gratis.

Copia y pega en Claude Code
/opsx:apply crear-proyectos

La IA ejecuta tasks.md línea por línea, marcando cada tarea al completarla. No intervengas salvo que te pregunte algo.

Con bun dev corriendo, en el navegador: crea 3 proyectos ("Casa", "Trabajo", "Estudio"), renombra uno, elimina otro y confirma que pide confirmación. Detén y vuelve a arrancar la app: los proyectos deben seguir ahí (eso es SQLite persistiendo en db.sqlite).

Copia y pega en Claude Code
/opsx:archive crear-proyectos

Mira ahora openspec/specs/: el plano maestro ya describe cómo funcionan los proyectos. La carpeta del cambio se movió al archivo histórico. Cierra con un commit:

git add -A && git commit -m "Módulo 1: proyectos"
✓ Checkpoint 1: proyectos funcionando + specs actualizadas + commit hecho. Limpia el contexto (/clear) antes del módulo 2.

Objetivo del módulo: cada proyecto tiene tareas, y cada tarea un estado: pendiente → en curso → hecha. Aquí verás la ventaja de OpenSpec en modo brownfield: la IA lee las specs del módulo 1 y construye sobre lo existente sin romperlo.

Copia y pega en Claude Code
/opsx:propose tareas-con-estado

Agregar tareas a los proyectos existentes:
- Una tarea pertenece a un proyecto y tiene: título (obligatorio) y estado.
- Estados posibles: "pendiente", "en curso", "hecha". Nace en "pendiente".
- Al seleccionar un proyecto en la barra lateral, veo sus tareas en el panel principal.
- Puedo: crear tarea, cambiar su estado con un clic (cicla entre los 3), editar el título y eliminarla.
- Filtro arriba del panel: Todas / Pendientes / En curso / Hechas.
- Al eliminar un proyecto, se eliminan también sus tareas.

Esta vez fíjate en algo nuevo: el delta. En specs/, OpenSpec marca qué requisitos se agregan y cuáles se modifican respecto al plano maestro. Busca:

  • Que la eliminación de proyectos ahora incluya el borrado de sus tareas (esto modifica un requisito del módulo 1 — es normal y está bien que aparezca marcado).
  • Que los 3 estados estén escritos exactamente como los pediste.
  • Que exista el filtro con las 4 opciones.
Si algo falta: no lo dejes pasar "porque seguro la IA lo arregla después". Pídelo ahora, en el documento. Lo que no está en la spec, no existe.

Mismo ritmo que el módulo 1, ahora sin guía detallada:

  1. /opsx:apply tareas-con-estado
  2. Prueba: crea tareas en dos proyectos distintos, cicla estados, usa el filtro, elimina un proyecto y verifica que sus tareas desaparecen.
  3. /opsx:archive tareas-con-estado y git commit.
✓ Checkpoint 2: tareas con estado + filtro funcionando, specs archivadas, commit hecho, contexto limpio.

Objetivo del módulo: registrar personas y asignarlas a tareas. Este módulo es deliberadamente menos guiado: el prompt de propose lo escribes tú. Requisitos que debe cubrir tu propuesta:

  • Una persona tiene nombre. Se pueden crear y listar (una pantalla o modal simple de "Equipo").
  • Una tarea puede tener una persona asignada (o ninguna).
  • En la lista de tareas se ven las iniciales de la persona asignada en un círculo.
  • Nuevo filtro: ver solo las tareas de una persona.
  • Si se elimina una persona, sus tareas quedan "sin asignar" (no se borran).
Plantilla — complétala con tus palabras
/opsx:propose personas-y-asignacion

[Describe aquí los 5 requisitos de arriba con tus propias palabras.
Sé concreto: qué datos tiene una persona, qué pasa al asignar,
qué se ve en pantalla, y qué pasa al eliminar una persona.]

Antes de aprobar, usa la lista de requisitos como checklist contra specs/. El punto 5 (eliminar persona sin borrar tareas) es el que la IA más tiende a resolver distinto a lo pedido: verifícalo con lupa.

  1. /opsx:apply personas-y-asignacion
  2. Prueba: crea 2 personas, asigna tareas, filtra por persona, elimina una persona y verifica que sus tareas quedan sin asignar.
  3. /opsx:archive personas-y-asignacion y git commit.
✓ Checkpoint 3: la app completa funciona y openspec/specs/ describe fielmente todo lo construido. Ábrelo y léelo: ese documento es tu aplicación explicada en español.

Hiciste 3 ciclos completos. Fíjate en lo que no hiciste: nunca escribiste código, nunca perdiste el hilo en un chat infinito, y en cualquier momento puedes explicar qué hace la app leyendo openspec/specs/.

Reto final (sin guía)

Agrega por tu cuenta, con un ciclo completo de OpenSpec, una de estas funciones:

  • Fecha límite en las tareas, con las vencidas resaltadas en rojo.
  • Contador de tareas pendientes junto a cada proyecto en la barra lateral.
  • Una tarea puede tener varias personas asignadas (¡ojo: esto modifica specs existentes!).

Criterio de éxito: que otra persona pueda entender tu cambio leyendo solo la carpeta archivada, sin mirar el código.

Lo que no está en la spec, no existe.
